# Context-Constrained AI Research in Transportation Engineering

## 1. 研究背景与动机

现有基于 Agent 的 AI Scientist 类研究（如 The AI Scientist, MLAgentBench）已展示了 LLM 驱动的 Agent 自主开展科学研究的潜力。然而，这些系统在处理复杂研究问题时频繁遇到性能瓶颈，其中一个核心但尚未被系统研究的限制因素是 **LLM 的上下文窗口（context window）**。

上下文窗口不仅限制了 Agent 单次能处理的信息量，更深刻地影响了 Agent 进行**多轮迭代试错**的能力——而这恰恰是科学研究的核心过程。当 Agent 需要在已有实验结果的基础上规划下一步实验时，历史信息的积累会不断压缩可用于当前任务的上下文空间。

本研究提出一个形式化框架，将 AI Agent 的自主实验过程建模为**上下文约束下的树搜索问题**，通过量化不同类型交通工程研究问题的上下文需求，分析在给定上下文窗口下哪些问题可以被 AI 有效解决、哪些仍需额外的工程优化。

**研究范围说明**：完整的 AI 自主科研可分解为三个阶段：(1) Idea 生成，(2) 自主实验，(3) 论文撰写。本研究聚焦阶段 (2)——自主实验，不讨论创新性和写作质量，仅关注 Agent 能否完成给定的研究任务。

---

## 2. 研究问题与假设

### 2.1 核心研究问题

**RQ**: 给定 LLM 的上下文窗口大小 $C$，AI Agent 能够自主完成哪些类型和复杂度的交通工程研究问题？其能力边界可否被理论预测？

### 2.2 研究假设

**H1（深度上界假设）**：对于给定的研究问题类型 $\mathcal{P}$ 和上下文窗口 $C$，Agent 能够执行的最大连续实验次数（研究深度）存在一个可计算的理论上界 $d^*$，且 Agent 的实际表现在接近 $d^*$ 时显著下降。

**H2（问题可分性假设）**：不同类型的交通研究问题具有不同的最小上下文需求 $C_{\min}$，据此可以将问题分为"当前可解"和"当前不可解"两类。

**H3（压缩差距假设）**：理想压缩下的理论研究深度与实际 LLM summarization 下的研究深度之间存在显著差距，该差距（压缩比 $\alpha$）是限制实际研究深度的主导因素。

## 3. 形式化模型：上下文约束树搜索（Context-Constrained Tree Search, CCTS）

### 3.1 基本定义

**定义 1（研究问题）**：研究问题 $\mathcal{P} = (O, \mathcal{D}, \mathcal{M}, \epsilon)$，其中 $O$ 为研究目标，$\mathcal{D}$ 为数据集，$\mathcal{M}$ 为可选方法空间，$\epsilon$ 为成功阈值（如 MAE $\leq \epsilon$）。

**定义 2（研究树）**：研究树 $\mathcal{T} = (V, E, r)$ 是一棵以 $r$ 为根的有根树：

- 根 $r$：初始问题设定（尚未进行任何实验）
- 节点 $v \in V$：一次具有具体结果的实验（包含实验配置和定量结果）
- 边 $(u, v) \in E$：基于节点 $u$ 的结果，决策执行新实验 $v$
- 深度 $d(v)$：从根到 $v$ 的边数
- 路径 $\text{Path}(v) = (r, v_1, v_2, \ldots, v)$：根到 $v$ 的唯一路径

**定义 3（实验结果）**：节点 $v$ 的实验结果为 $R(v) = (\text{config}(v), \text{metric}(v), \text{status}(v))$，其中 $\text{status} \in \text{success}, \text{fail}, \text{error}$。

**定义 4（研究成功）**：若存在节点 $v \in V$ 使得 $\text{metric}(v) \leq \epsilon$（或满足特定成功条件），则称问题 $\mathcal{P}$ 在树 $\mathcal{T}$ 上被成功解决。

### 3.2 上下文分配模型

在节点 $v$ 处启动的 subagent，其上下文窗口 $C$ 被分配为四部分：

$$C = \underbrace{c_{\text{sys}} + c_{\text{prob}}(\mathcal{P})}*{S(\mathcal{P}) \text{：静态开销}} + \underbrace{c*{\text{hist}}(v)}*{\text{继承上下文}} + \underbrace{c*{\text{work}}(v)}_{\text{有效工作上下文}}$$

各分量定义：


| 分量                             | 含义                      | 决定因素     | 性质       |
| ------------------------------ | ----------------------- | -------- | -------- |
| $c_{\text{sys}}$               | Agent 系统指令、工具描述、行为规范    | Agent 框架 | 常数，与问题无关 |
| $c_{\text{prob}}(\mathcal{P})$ | 问题定义：目标、数据描述、约束、评价标准    | 问题类型     | 与问题相关的常数 |
| $c_{\text{hist}}(v)$           | 从根到 $v$ 父节点的所有祖先实验的压缩摘要 | 树中位置     | 随深度单调递增  |
| $c_{\text{work}}(v)$           | 执行当前实验可用的剩余上下文          | 上述三者的剩余  | 随深度单调递减  |


有效工作上下文：

$$c_{\text{work}}(v) = C - S(\mathcal{P}) - c_{\text{hist}}(v)$$

### 3.3 任务上下文需求

执行节点 $v$ 处的实验所需的最小上下文为 **任务上下文需求** $D$：

$$D(\mathcal{P}, v) = c_{\text{code}}(\mathcal{P}) + c_{\text{env}}(\mathcal{P}) + c_{\text{cot}}(\mathcal{P}, v) + c_{\text{out}}(\mathcal{P}, v)$$


| 子分量                              | 含义                               | 主要决定因素                |
| -------------------------------- | -------------------------------- | --------------------- |
| $c_{\text{code}}(\mathcal{P})$   | 完成实验需要读写的代码                      | 问题类型（数据加载、模型定义、训练、评估） |
| $c_{\text{env}}(\mathcal{P})$    | 环境/工具的上下文（API 文档、数据 schema、文件结构） | 问题所依赖的外部工具            |
| $c_{\text{cot}}(\mathcal{P}, v)$ | Agent 的推理过程（规划、调试、决策）            | 问题难度 + 模型推理能力         |
| $c_{\text{out}}(\mathcal{P}, v)$ | 实验产生的输出（训练日志、评估指标、错误信息）          | 实验配置（如训练轮数、日志详细程度）    |


**简化假设**：对于同一问题类型 $\mathcal{P}$ 的不同节点，任务上下文需求近似为常数 $D(\mathcal{P})$。

**节点可行性条件**：

$$c_{\text{work}}(v) \geq D(\mathcal{P}) \quad \Longleftrightarrow \quad C - S(\mathcal{P}) - c_{\text{hist}}(v) \geq D(\mathcal{P})$$

### 3.4 理想压缩假设

**假设（Ideal Compression）**：存在理想化的压缩算法，能将已知信息压缩至其信息论下界，即压缩后的上下文长度等于该信息的最短描述长度。

在此假设下：

$$c_{\text{hist}}(v) = \sum_{u \in \text{Path}(v) \setminus v} \Delta I(u)$$

其中 $\Delta I(u)$ 为节点 $u$ 的 **信息增量**——该次实验结果的最短描述长度，包括：

- 实验配置的关键参数
- 定量结果指标
- 定性结论（如适用）

设所有节点的平均信息增量为 $\overline{\Delta I}$，则深度为 $d$ 的节点：

$$c_{\text{hist}}(v) \approx d \cdot \overline{\Delta I}$$

**实际压缩**：引入压缩比 $\alpha \geq 1$，实际 LLM summarization 的输出长度为：

$$c_{\text{hist}}^{\text{actual}}(v) = \alpha \cdot c_{\text{hist}}^{\text{ideal}}(v) = \alpha \cdot d \cdot \overline{\Delta I}$$

当 $\alpha = 1$ 时为理想压缩，$\alpha > 1$ 反映实际 summarization 与理想压缩的差距。

### 3.5 最大研究深度

**定理 1（最大研究深度）**：给定上下文窗口 $C$、研究问题 $\mathcal{P}$、压缩比 $\alpha$，Agent 能执行的最大连续实验次数为：

$$d^*(\mathcal{P}, C, \alpha) = \left\lfloor \frac{C - S(\mathcal{P}) - D(\mathcal{P})}{\alpha \cdot \overline{\Delta I}} \right\rfloor$$

**推论 1（理想深度 vs 实际深度）**：

$$d^*_{\text{ideal}} = \left\lfloor \frac{C - S(\mathcal{P}) - D(\mathcal{P})}{\overline{\Delta I}} \right\rfloor, \quad d^{\text{actual}} = \left\lfloor \frac{d^**{\text{ideal}}}{\alpha} \right\rfloor$$

即实际研究深度约为理想深度的 $1/\alpha$。

### 3.6 研究问题可解性

**定义 5（最小实验次数）**：$k^*(\mathcal{P})$ 为解决问题 $\mathcal{P}$ 所需的最小实验次数（期望值），取决于方法空间 $\mathcal{M}$ 的大小和 Agent 的搜索效率。

**定理 2（可解性条件）**：问题 $\mathcal{P}$ 在上下文窗口 $C$ 和压缩比 $\alpha$ 下可解，当且仅当：

$$d^*(\mathcal{P}, C, \alpha) \geq k^*(\mathcal{P})$$

等价地，解决问题所需的最小上下文窗口为：

$$C_{\min}(\mathcal{P}, \alpha) = S(\mathcal{P}) + D(\mathcal{P}) + k^*(\mathcal{P}) \cdot \alpha \cdot \overline{\Delta I}$$

**物理含义**：$C_{\min}$ 由三部分构成：

1. $S(\mathcal{P})$：问题的"入场费"——仅仅理解问题就需要的上下文
2. $D(\mathcal{P})$：问题的"工作台大小"——执行一次实验需要的空间
3. $k^* \cdot \alpha \cdot \overline{\Delta I}$：问题的"记忆成本"——记住所有历史实验所需的空间

## 4. 实验设计

### 4.1 Phase 1：交通预测案例研究

#### 4.1.1 问题定义

$\mathcal{P}_{\text{pred}}$：基于图神经网络的交通速度预测


| 要素                 | 设定                                           |
| ------------------ | -------------------------------------------- |
| 目标 $O$             | 预测路网中各检测器未来 T 步的交通速度                         |
| 数据集 $\mathcal{D}$  | METR-LA 或 PEMS-BAY（标准 benchmark）             |
| 方法空间 $\mathcal{M}$ | GCN / GAT / STGCN / DCRNN / Transformer 及其变体 |
| 成功阈值 $\epsilon$    | MAE 达到已发表 baseline 水平                        |


#### 4.1.2 上下文参数的理论估算

**静态开销 $S$**：


| 分量                | 估算依据                                                                          | 估计值（tokens） |
| ----------------- | ----------------------------------------------------------------------------- | ----------- |
| $c_{\text{sys}}$  | 典型 Agent system prompt（角色定义 + 行为规范 + 工具描述：如 shell、file read/write、python 执行等） | ~2,000      |
| $c_{\text{prob}}$ | 数据集说明（来源、维度、时间范围、空间结构）+ 任务定义 + 评价指标 + 可用工具/库约束                                | ~1,500      |
| **$S$ 合计**        |                                                                               | **~3,500**  |


**任务上下文需求 $D$**：


| 分量                | 估算依据                                                                 | 估计值（tokens） |
| ----------------- | -------------------------------------------------------------------- | ----------- |
| $c_{\text{code}}$ | 数据加载 (~~800) + 模型定义 (~~1,500) + 训练循环 (~~800) + 评估脚本 (~~400)          | ~3,500      |
| $c_{\text{env}}$  | PyTorch/PyG 关键 API 用法 + 数据文件 schema + 项目文件结构                         | ~1,000      |
| $c_{\text{cot}}$  | Agent 的规划（选择模型、设计实验）+ 调试推理（分析报错、修改代码）                                | ~3,000      |
| $c_{\text{out}}$  | 训练日志（每 epoch 一行 × 100 epochs ≈ 500）+ 最终评估指标 (~~100) + 潜在报错信息 (~~500) | ~1,100      |
| **$D$ 合计**        |                                                                      | **~8,600**  |


**信息增量 $\overline{\Delta I}$**：


| 压缩方式      | 估算依据                                                                     | 估计值（tokens） |
| --------- | ------------------------------------------------------------------------ | ----------- |
| 理想压缩      | "STGCN, hidden=64, lr=1e-3 → MAE=4.23, RMSE=6.15, converged at epoch 80" | ~50         |
| 实际 LLM 摘要 | 包含实验动机、方法描述、结果分析、后续建议等                                                   | ~400-800    |


由此估算压缩比 $\alpha \approx 8 \sim 16$。

#### 4.1.3 理论预测

以 128K token 上下文为例：


| 指标    | 理想压缩（$\alpha=1$）                             | 实际压缩（$\alpha=10$）                            |
| ----- | -------------------------------------------- | -------------------------------------------- |
| $d^*$ | $\lfloor(128000-3500-8600)/50\rfloor = 2318$ | $\lfloor(128000-3500-8600)/500\rfloor = 231$ |
| 含义    | 理论上可做 2318 次实验                               | 实际可做约 231 次实验                                |


以 32K token 上下文为例：


| 指标    | 理想压缩（$\alpha=1$）                           | 实际压缩（$\alpha=10$）                          |
| ----- | ------------------------------------------ | ------------------------------------------ |
| $d^*$ | $\lfloor(32000-3500-8600)/50\rfloor = 398$ | $\lfloor(32000-3500-8600)/500\rfloor = 39$ |
| 含义    | 理论上可做 398 次实验                              | 实际可做约 39 次实验                               |


#### 4.1.4 实验方案

**实验 1：参数实测**

- 使用一个具体的 Agent 框架（如基于 GPT-4o / Claude 的自定义 Agent）
- 让 Agent 执行交通预测任务，记录每次实验中各上下文分量的实际 token 数
- 与理论估算值对比，校准模型参数

**实验 2：深度极限测试**

- 控制 Agent 的可用上下文窗口（如分别设为 32K、64K、128K）
- 让 Agent 自主迭代实验，记录：
  - 每个节点的 $c_{\text{hist}}$、$c_{\text{work}}$ 的实际值
  - Agent 在不同深度的实验成功率
  - Agent 何时开始出现性能退化（无法完成实验、代码错误增多等）
- 验证 H1：实际退化点是否接近理论 $d^*$

**实验 3：压缩比测量**

- 对同一组实验结果，分别采用：
  - 理想压缩：人工撰写最短描述
  - LLM 摘要：让模型自行总结
  - 结构化摘要：预定义模板（JSON 格式：配置 + 指标）
- 测量三种方式的 token 数，计算实际压缩比 $\alpha$
- 验证 H3

**实验 4：问题难度梯度**

- 在交通预测问题内部设置难度梯度：
  - Level 1：单步预测，简单 MLP baseline
  - Level 2：多步预测，需要时空建模（GCN）
  - Level 3：多步预测 + 多数据集泛化，需要架构搜索
- 估算各难度级别的 $k^*$，验证 H2

### 4.2 Phase 2：跨问题泛化

在 Phase 1 验证框架有效性后，扩展至另外两类问题：

#### 4.2.1 网络均衡分析（UE/SO）


| 要素   | 设定                                                                  |
| ---- | ------------------------------------------------------------------- |
| 目标   | 实现用户均衡/系统最优分配算法并在给定路网上求解                                            |
| 方法空间 | Frank-Wolfe / MSA / 梯度投影 / Column Generation                        |
| 特点   | $c_{\text{code}}$ 低, $c_{\text{cot}}$ 高（需要数学推理）, $c_{\text{env}}$ 低 |
| 预期   | $D$ 较小, $C_{\min}$ 最低, 最容易被 AI 解决                                   |


#### 4.2.2 信号灯优化（RL）


| 要素   | 设定                                                             |
| ---- | -------------------------------------------------------------- |
| 目标   | 用强化学习优化交叉口信号配时，最小化平均延误                                         |
| 方法空间 | DQN / PPO / A2C + 不同 reward 设计 + 不同状态表示                        |
| 特点   | $c_{\text{env}}$ 极高（SUMO API + 路网配置）, $c_{\text{out}}$ 高（训练日志） |
| 预期   | $D$ 最大, $C_{\min}$ 最高, 对 AI 最具挑战性                              |


#### 4.2.3 跨问题比较

核心目标：验证框架预测的 $C_{\min}$ 排序是否与实际 Agent 表现一致。

预期排序：$C_{\min}(\text{equilibrium}) < C_{\min}(\text{prediction}) < C_{\min}(\text{signal})$

### 4.3 评估指标体系


| 指标                           | 定义                                                                 | 用途          |
| ---------------------------- | ------------------------------------------------------------------ | ----------- |
| 理论研究深度 $d^*_{\text{ideal}}$  | $\lfloor(C-S-D)/\overline{\Delta I}\rfloor$                        | 理论上界        |
| 实际研究深度 $d^*_{\text{actual}}$ | Agent 在实验中实际达到的最大深度                                                | 实测值         |
| 压缩比 $\alpha$                 | $c_{\text{hist}}^{\text{actual}} / c_{\text{hist}}^{\text{ideal}}$ | 量化压缩差距      |
| 深度-成功率曲线                     | 不同深度下实验成功率 $P(d)$                                                  | 验证 H1       |
| 最小上下文 $C_{\min}$             | 使问题可解的最小上下文窗口                                                      | 问题可解性分类     |
| 上下文利用效率                      | $\text{metric improvement} / \text{tokens consumed}$               | 衡量 Agent 效率 |

## 5. 预期贡献

1. **理论框架**：首次提出将 AI 自主科研过程建模为上下文约束树搜索问题的形式化框架（CCTS），为分析 AI Agent 的科研能力边界提供理论工具。
2. **可解性分析方法**：提出基于 $C_{\min}$ 的研究问题可解性分析方法，为交通工程领域评估"哪些研究可以交给 AI"提供量化依据。
3. **压缩差距的量化**：首次实测理想压缩与实际 LLM summarization 之间的差距 $\alpha$，为上下文管理的工程优化指明方向。
4. **交通问题分类**：基于上下文需求对典型交通研究问题进行分类，建立"AI 可解性图谱"。

## 6. 潜在局限与未来方向

### 当前框架的简化假设

- **单路径模型**：当前模型仅考虑树的单条路径上的上下文积累，未建模跨分支的信息共享（如"分支 A 失败了"的信息对分支 B 的价值）
- **固定需求假设**：假设 $D(\mathcal{P})$ 对同一问题类型为常数，实际上不同实验的复杂度可能差异较大
- **忽略注意力退化**（机制 B）：未建模 "Lost in the Middle" 等注意力退化效应，此效应可能使实际性能在 $d^*$ 之前就开始下降

### 未来方向

- **机制 B 的建模**：引入上下文长度-准确率衰减函数 $f(l)$，使成功概率 $P(v) = g(c_{\text{work}}(v), c_{\text{hist}}(v))$ 同时依赖于可用空间和已占用空间
- **Idea 生成与论文撰写阶段的扩展**：将 CCTS 框架扩展至完整的 AI 科研流程
- **跨领域泛化**：将框架应用于交通以外的工程领域（如材料科学、药物发现）
- **工程优化指导**：基于框架分析结果，指导上下文管理策略的设计（如动态压缩、选择性遗忘、外部记忆增强等）
